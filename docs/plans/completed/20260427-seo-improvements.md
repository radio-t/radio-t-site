# SEO Improvements for radio-t.com

## Overview

Google Search Console export (last 90 days) shows the radio-t.com Hugo site has solid query volume (92,083 impressions) but weak CTR on desktop (1.54%) and on every page except the homepage. Investigation of the Hugo templates revealed multiple concrete defects: no Open Graph tags, broken `twitter:description` (references undefined config key), site-wide identical `<meta description>` on every page, no JSON-LD structured data, indexed pagination pages with wrong canonical URLs, and a five-year-old Hugo (0.81.0) that is overdue for an upgrade.

This plan addresses every defect found in the templates and config. **No content files (`hugo/content/posts/*.md`) are modified** — improvements are entirely template- and config-driven, so future episode markdown stays unchanged. Episode-title enrichment (which would require touching the markdown front-matter) is explicitly scoped out as a separate future plan.

The Hugo upgrade from 0.81.0 to current is performed **first**, against the unchanged-template baseline, so the visual regression diff shows only Hugo-version-induced output differences (template-function changes, output formatter tweaks). Once Hugo is verified clean, SEO improvements are layered on top — any subsequent diff is then known to come from the new template tags, not from Hugo itself. This ordering is the only way to make the visual regression gate enforceable.

`agent-browser` is used after the Hugo bump to side-by-side compare local Hugo serve output between old and new versions on representative URLs.

### CRITICAL CONSTRAINT: RSS feeds must remain byte-identical

The primary podcast feeds (`https://radio-t.com/podcast.rss` and `https://radio-t.com/podcast-archives.rss`) are consumed by podcast clients (Apple Podcasts, Overcast, Pocket Casts, etc.). **Any change in feed contents — even cosmetic — risks triggering mass episode re-downloads across thousands of subscribers**. This is unacceptable.

Risk surface:
- `podcast.rss` and `podcast-archives.rss` are generated by the Go binary `rss_generator` (in `radio-t-site/rss_generator/`), invoked from `exec.sh` after `hugo --minify`. The Go binary reads markdown directly and does not depend on Hugo's template engine. **These feeds are structurally unaffected by the Hugo bump.** They MUST be diffed byte-for-byte regardless, as a sanity check.
- Hugo also auto-generates RSS feeds at `/index.xml`, `/posts/index.xml`, `/categories/*/index.xml` (uses `hugo/layouts/rss.xml`). These can change between Hugo versions (date format, escaping, GUID rules). They have a smaller subscriber base but must still be tracked.
- SEO template changes (Tasks 2-7) only touch `head.html`, `single.html`, `list.html`, `index.html`, `jsonld.html`, `robots.txt` — none of these affect RSS output. RSS byte-identity must still be re-verified after Tasks 2-7 as a final guard.

Hard gate: the Hugo bump and the SEO changes are both blocked from merging until `podcast.rss` and `podcast-archives.rss` are confirmed byte-identical between baseline and new build. Hugo's auto-generated `*.xml` feeds may change with prior approval; if they change, document which feeds and the nature of the diff before proceeding.

## Technical Details

### Hreflang — explicitly NOT added

After review: radio-t.com is a single-language site (Russian only). Self-referential `hreflang="ru"` + `x-default` to the same URL provides no SEO value (Google uses hreflang only to disambiguate multi-language/region variants). Dropped to avoid adding bytes without benefit.

### JSON-LD schema choice — Article (BlogPosting) for posts, PodcastSeries for site

Google deprecated podcast-specific structured-data rich results; podcast surfaces in Google now derive from RSS feed discovery, not on-page JSON-LD. To keep Google rich-result eligibility, podcast posts emit `BlogPosting` (a recognized Article type) with `audio` as an embedded `MediaObject`. The site-level `PodcastSeries` schema is kept on the homepage — it remains valid schema.org and is consumed by non-Google crawlers, AI Overviews, and voice assistants. Both schemas are emitted via Hugo's `dict` builder + `jsonify` to guarantee valid JSON regardless of title/description content (Cyrillic, quotes, special characters).

### Description fallback chain (in head.html)

```go-html-template
{{ $desc := "" }}
{{ with .Description }}{{ $desc = . }}
{{ else }}{{ with .Summary }}{{ $desc = . | plainify | truncate 160 }}
{{ else }}{{ $desc = .Site.Params.description }}{{ end }}{{ end }}
```

Applied uniformly to `<meta name="description">`, `og:description`, and `twitter:description`.

### Canonical URL handling for paginated pages

Inside `head.html`, detect paginated state and emit the correct canonical/og:url. Approach:

```go-html-template
{{ $canonical := .Permalink }}
{{ with .Scratch.Get "paginator" }}
  {{ if gt .PageNumber 1 }}
    {{ $canonical = .URL | absURL }}
  {{ end }}
{{ end }}
```

The homepage's `index.html` already stores its paginator into `.Scratch` (verified: `{{ .Scratch.Set "paginator" $paginator }}` line 4 of `index.html`). Section list pages use `.Paginator` directly. The head partial reads from Scratch first, falls back to `.Permalink`.

### Open Graph tags (in head.html, all pages)

```
og:type        = website (homepage/list) | article (post)
og:title       = .Title
og:description = $desc (fallback chain above)
og:image       = .Params.image | site default cover image
og:image:width  = 400
og:image:height = 400
og:url         = $canonical (paginated-aware, see above)
og:site_name   = .Site.Title
og:locale      = ru_RU
```

Cover dimensions verified at 400×400 (`file rt1010.jpg`).

### Twitter Card

`summary` → `summary_large_image` site-wide. All `twitter:*` tags moved from `single.html` to `head.html`. Standardize URL on `.Permalink` (drop the `{{ .Site.Params.url }}{{ .RelPermalink }}` form).

### JSON-LD: PodcastSeries (homepage only)

Built via Hugo `dict` and emitted with `jsonify`:

```go-html-template
{{ if eq .RelPermalink "/" }}
{{ $series := dict
  "@context" "https://schema.org"
  "@type" "PodcastSeries"
  "name" .Site.Title
  "description" .Site.Params.description
  "url" .Site.Params.url
  "image" "https://radio-t.com/images/radio-t/cover.png"
  "inLanguage" "ru-RU"
  "webFeed" "https://radio-t.com/podcast.rss"
}}
<script type="application/ld+json">{{ $series | jsonify }}</script>
{{ end }}
```

### JSON-LD: BlogPosting + audio (every podcast post)

```go-html-template
{{ if in .Params.categories "podcast" }}
{{ $audio := dict
  "@type" "MediaObject"
  "contentUrl" (printf "https://cdn.radio-t.com/%s.mp3" .Params.filename)
  "encodingFormat" "audio/mpeg"
}}
{{ $post := dict
  "@context" "https://schema.org"
  "@type" "BlogPosting"
  "headline" .Title
  "url" .Permalink
  "datePublished" (.Date.Format "2006-01-02")
  "dateModified" (.Lastmod.Format "2006-01-02")
  "image" .Params.image
  "inLanguage" "ru-RU"
  "audio" $audio
  "publisher" (dict "@type" "Organization" "name" .Site.Title "url" .Site.Params.url)
}}
<script type="application/ld+json">{{ $post | jsonify }}</script>
{{ end }}
```

`jsonify` handles all string escaping; Cyrillic, quotes, and HTML entities in titles/descriptions are safe.

### Pagination noindex

The homepage paginator lives in `index.html` and is exposed via `.Scratch.Set "paginator"`. Section list pages use `.Paginator` directly. In both `index.html` and `list.html`, define a `head` block:

```go-html-template
{{ define "head" }}
  {{ $pn := 1 }}
  {{ with .Scratch.Get "paginator" }}{{ $pn = .PageNumber }}{{ end }}
  {{ with .Paginator }}{{ if gt .PageNumber 1 }}{{ $pn = .PageNumber }}{{ end }}{{ end }}
  {{ if gt $pn 1 }}<meta name="robots" content="noindex, follow">{{ end }}
{{ end }}
```

`baseof.html` already has `{{ block "head" . }}{{ end }}` (verified line 6), so this composes cleanly.
